Application Domain: Home Automation

I've been running Home Assistant for a while now and it's one of those projects I genuinely enjoy tinkering with. I have automations that coordinate sleep routines between multiple people, adjust lights based on who's home and what time of day it is, manage climate control when it's hot out, and even run weekly system updates that loop through every addon. At this point my setup has real complexity to it, with variables, conditionals, loops, and reusable routines. So when this assignment asked us to design a no-code programming environment, home automation was an easy pick. I already know where the pain points are.

The biggest one is how you actually build automations. Home Assistant gives you two options: a visual drag-and-drop editor or raw YAML code. The visual editor is great for simple stuff, but it falls apart with complex logic. At that point you switch to YAML, and once you do, you lose the visual view entirely. Switching back often breaks things. You're stuck choosing one or the other, and neither is great on its own.

AutomateHome is my take on fixing this. It's a visual programming environment where you build automations by dragging and connecting blocks, but with one difference that changes everything: the code view is always right there next to the canvas, updating in real time as you work.

The Big Idea: Side-by-Side Code Sync

As you drag a block onto the canvas, the YAML on the right updates instantly. Click a block and the corresponding lines highlight in the code. Edit the YAML directly and the blocks rearrange to match. This is powerful for a few reasons.

It helps you learn the code. Most people start with the visual editor because it's approachable. But as their automations grow, they eventually need to understand the code. Instead of a hard cutover from visual to code, the side-by-side view lets you watch the code take shape as you drag blocks around. You start to recognize patterns. "Oh, that's what a condition looks like in YAML." Over time, you naturally pick up the syntax without ever sitting down to study it.

It gives you code-level control when you need it. Sometimes the visual editor is faster. Sometimes typing a line of YAML is faster. With the sync view, you don't have to choose. Drag a block to set up the basic structure, then click into the YAML to fine-tune a specific value. Both sides stay in sync, so there's no risk of one overwriting the other.

It makes debugging easier. When an automation isn't working, being able to see both the visual flow and the underlying code at the same time helps you figure out what went wrong. You can trace through the flow visually while checking the actual values in the code.

This is Shneiderman's direct manipulation in action: continuous representation (the automation is always visible in both forms), physical actions instead of syntax (drag blocks to build logic), and immediate feedback (every change shows up instantly in both views).

Loops are one of the things the existing HA visual editor can't represent. You end up writing Jinja2 templates in YAML. Here, you just drag a "For Each" block and drop actions inside it. The dashed border makes it clear which actions repeat and which don't. The loop variable ("current item") is available to everything inside the border.

The schedule trigger supports daily, weekly, monthly, and custom cron expressions, configured through a calendar picker instead of typed syntax.

Parameters are shown as pink blocks at the top of the routine. Inside, actions can reference these parameters. The "If Temperature > ac_temp_threshold" condition uses the parameter value, so one person can set it to 70 and the other to 72 when calling the same routine. If you update the routine, the change automatically applies everywhere it's used.
